# [3286. Find a Safe Walk Through a Grid](https://leetcode.com/problems/find-a-safe-walk-through-a-grid)

When navigating complex grids in competitive programming, the challenge often isn't just finding *a* path, but finding the *best* path under specific constraints—such as maintaining a certain resource level (like health) throughout the journey.

In this post, we will break down the problem **"Find a Safe Walk Through a Grid"** and look at an efficient way to solve it using a Best-First Search approach, similar to Dijkstra's algorithm.

## The Problem Breakdown

The objective is to determine if there exists a path from the top-left cell $(0, 0)$ to the bottom-right cell $(m-1, n-1)$ such that your health remains strictly greater than zero at every step.

**Constraints:**
* You start with a given `health`.
* Each cell contains a value that adds to your current health.
* You can move in four directions: Up, Down, Left, and Right.
* You must arrive at the destination with `health > 0`.

## The Approach: Best-First Search (Dijkstra Variation)

A straightforward Breadth-First Search (BFS) might not be sufficient because we aren't just looking for the shortest path in terms of steps, but the "safest" path in terms of accumulated health. 

To solve this, we can treat this as a variation of the shortest path problem. Instead of minimizing distance, we want to find a path that minimizes the "drain" on our health. 

### The Strategy
1. **The Greedy Choice:** We use a **Priority Queue** to always explore the cell that is currently "safest" (the one that keeps our cumulative health offset the lowest).
2. **The Offset Trick:** To make the implementation elegant, we track the `current_offset`, which is calculated as `(Sum of grid values encountered) - (Initial Health)`. 
   * If this offset remains negative, it means our actual health is still positive.
   * By using a Min-Heap (Priority Queue) on this offset, we ensure we are always exploring the path that leaves us with the most potential health.
3. **State Management:** We keep track of `visited` cells to ensure we don't enter infinite loops and to optimize the search.

## Implementation

Here is the refined Python solution. I have cleaned up the variable names to reflect the "offset" logic, making the code much easier for readers to follow while maintaining your original, clever logic.

```python
from queue import PriorityQueue
from typing import List

class Solution:
    def findSafeWalk(self, grid: List[List[int]], health: int) -> bool:
        m, n = len(grid), len(grid[0])
        
        def is_valid_cell(r, c):
            return 0 <= r < m and 0 <= c < n

        # Directions: Right, Down, Left, Up
        directions = [(0, 1), (1, 0), (0, -1), (-1, 0)]
        
        # pq will store tuples of (current_offset, row, col)
        # current_offset = (Sum of cells visited) - initial_health
        pq = PriorityQueue()
        
        # Initial state: Start at (0,0). 
        # The offset at start is grid[0][0] - initial_health
        initial_offset = grid[0][0] - health
        pq.put((initial_offset, 0, 0))
        
        visited = set()
        visited.add((0, 0))
        
        while not pq.empty():
            current_offset, r, c = pq.get()
            
            # If we reached the destination and our offset is still negative,
            # it means our health (health - cumulative_sum) is > 0.
            if current_offset < 0 and r == m - 1 and c == n - 1:
                return True
            
            # We only continue exploring if we are still "alive" (offset < 0)
            if current_offset < 0:
                for dr, dc in directions:
                    nr, nc = r + dr, c + dc
                    
                    if is_valid_cell(nr, nc) and (nr, nc) not in visited:
                        visited.add((nr, nc))
                        # The new offset is the current offset + the value of the next cell
                        new_offset = current_offset + grid[nr][nc]
                        pq.put((new_offset, nr, nc))
                        
        return False

```

## Complexity Analysis

* **Time Complexity:** $O(M \cdot N \log(M \cdot N))$
  In the worst case, we visit every cell in the $M \times N$ grid. Each insertion and extraction from the Priority Queue takes $O(\log(M \cdot N))$ time.
  
* **Space Complexity:** $O(M \cdot N)$
  We use a `visited` set and a `PriorityQueue` that can store up to $M \times N$ elements in the worst case.

## Summary
By reframing the "health" problem into a "cumulative offset" problem, we can utilize Dijkstra's algorithm to greedily find the safest possible path. This approach ensures that if a valid path exists, we will find it efficiently by always prioritizing the path that preserves the most health.
