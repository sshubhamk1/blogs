# [3620. Network Recovery Pathways](https://leetcode.com/problems/network-recovery-pathways)

## The Challenge

At first glance, this problem appears to be a variation of the Shortest Path problem. However, it is more complex because we are juggling two distinct objectives:
1.  **Maximizing the "bottleneck":** We want the weakest link in our path to be as strong as possible.
2.  **Adhering to a "budget":** The total cost (sum of edge weights) cannot exceed $k$.

A naive approach—such as exploring all possible paths using Depth First Search (DFS)—would be inefficient. Since the number of paths in a graph can grow exponentially with the number of nodes and edges, a brute-force search will inevitably result in a **Time Limit Exceeded (TLE)** error.

## The Optimal Approach: Binary Search on the Answer + Dijkstra

The core realization needed to solve this problem efficiently is **monotonicity**. 

If it is possible to find a path from $0$ to $n-1$ where every edge has a weight of at least $X$ and the total weight sum is $\le k$, then it is guaranteed that we can also find such a path for any value $Y$ where $Y < X$. This monotonic property is the "green light" to use **Binary Search on the Answer**.

### Strategy

1.  **Binary Search Range:** The potential "minimum weight" we are looking for lies between `0` and the maximum possible edge weight (which we can cap at $k$).
2.  **The Decision Function (`is_possible`):** For a specific `target` weight, we need to answer the question: *"Can we reach node $n-1$ from node $0$ using only edges with $\text{weight} \ge \text{target}$ such that the total path weight remains $\le k$?"*
3.  **Shortest Path Verification:** To answer the decision function efficiently, we use **Dijkstra's Algorithm**. For a fixed `target`, we filter the graph:
    *   Only consider edges where $\text{weight} \ge \text{target}$.
    *   Only consider nodes where $\text{node\_status} == \text{online}$.
    *   Dijkstra's will compute the *minimum possible total weight* to reach the destination under these constraints.
4.  **Final Decision:** 
    *   If Dijkstra's finds a shortest path $\leq k$, then the `target` is feasible. We record this as a potential answer and try a larger weight (move `left` pointer up).
    *   Otherwise, the `target` is too restrictive. We try a smaller weight (move `right` pointer down).

## Implementation

```python
import heapq
import collections
import math
from typing import List

class Solution:
    def findMaxPathScore(self, edges: List[List[int]], online: List[bool], k: int) -> int:
        """
        Finds the maximum possible minimum edge weight in a path from 0 to n-1
        subject to a total weight constraint k and online node availability.
        """
        n = len(online)
        
        # Corner Case: If start or end node is offline, no valid path exists.
        if not online[0] or not online[n-1]:
            return -1
            
        # Build adjacency list: graph[u] = [(v, weight), ...]
        graph = collections.defaultdict(list)
        for u, v, weight in edges:
            graph[u].append((v, weight))

        def is_path_possible(min_allowed_weight: int) -> bool:
            """
            Uses Dijkstra's algorithm to determine if a path exists from 0 to n-1
            where every edge weight >= min_allowed_weight and total_sum <= k.
            """
            # min_dist[i] stores the minimum weight sum to reach node i
            min_dist = [math.inf] * n
            min_dist[0] = 0
            
            # Priority queue stores (current_total_weight, current_node)
            pq = [(0, 0)]
            
            while pq:
                current_weight_sum, u = heapq.heappop(pq)
                
                # Optimization: If we've already found a better path to u, skip.
                if current_weight_sum > min_dist[u]:
                    continue
                
                # Optimization: If we reached the destination within budget, return True.
                if u == n - 1:
                    return current_weight_sum <= k
                
                for v, weight in graph[u]:
                    # Check constraints:
                    # 1. Destination node v must be online.
                    # 2. Edge weight must meet our binary search target.
                    if online[v] and weight >= min_allowed_weight:
                        new_dist = current_weight_sum + weight
                        
                        # 3. The total path sum must not exceed our budget k.
                        if new_dist <= k and new_dist < min_dist[v]:
                            min_dist[v] = new_dist
                            heapq.heappush(pq, (new_dist, v))
                            
            return min_dist[n-1] <= k

        # Binary search for the maximum possible "minimum weight"
        left, right = 0, k
        max_min_weight = -1
        
        while left <= right:
            mid = (left + right) // 2
            if is_path_possible(mid):
                # This weight is achievable; attempt to find a larger minimum weight.
                max_min_weight = mid  
                left = mid + 1
            else:
                # Target weight is too high; decrease the threshold.
                right = mid - 1
                
        return max_min_weight
```

## Complexity Analysis

### Time Complexity
* **Binary Search:** The search space is the range of weights, which is bounded by $k$. This contributes $O(\log(k))$ steps.
* **Dijkstra's Algorithm:** For every step in the binary search, we execute Dijkstra's algorithm. Dijkstra's complexity is $O(E \log V)$, where $E$ is the number of edges and $V$ is the number of nodes.
* **Total Complexity:** $O(\log(k) \cdot E \log V)$. This approach is highly efficient even for large graphs.

### Space Complexity
* **Adjacency List:** $O(V + E)$ to store the graph structure.
* **Dijkstra's Data Structures:** $O(V)$ for the distance array and the priority queue.
* **Total Complexity:** $O(V + E)$.

## Key Takeaway
This problem is a classic implementation of the **"Binary Search on the Answer"** paradigm. Whenever a problem asks you to **"maximize the minimum"** or **"minimize the maximum"**, check if the problem exhibits monotonicity. If it does, you can transform a complex optimization problem into a series of simpler feasibility checks, significantly reducing the computational overhead.
